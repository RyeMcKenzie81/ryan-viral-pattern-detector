## Problem Statement

ViralTracker has 82 services but only 20 are accessible through the Chainlit chat interface. The other 62 are locked behind 78 Streamlit pages with rigid, pre-built workflows. (Breakdown: 20 with agent tools, 11 instantiated in AgentDeps but no tools, 51 UI-only.) The founder runs an agency generating $20K+/month in revenue using these tools, but constantly hits walls where the UI can't compose services in the ways the moment demands.

Examples of blocked workflows:
- "Take top 10 ROAS ads for March, recreate with a different persona, 3 variations each with new hooks"
- "Find linking opportunities between our articles for a target keyword, then execute the changes"
- "What's the aggregate cost per add-to-cart by consumer awareness level?"
- "When was this winning ad published? How many purchases? Is it scaling or stuck?"

The services exist for all of these. The UI doesn't expose them in the right combinations.

## Demand Evidence

The founder IS the primary user. Direct quotes:

- "I use it to run agency work. But it feels like things constantly break and when I need to use things I need to go and update code because something else broke another thing."
- "Sometimes I want to ask it to do something that I know we have services that can check, but our Streamlit interface doesn't have the features to deviate from what UI is built."
- "Right now I want to know when this ad was published, how many purchases we've had, so I can start to understand if this is a new ad that needs more time, or if it's just serving to a very small audience."

Revenue at risk: $20K+/month in agency services. Every minute fighting the UI is time not spent on client work.

## Status Quo

Current workaround when the UI doesn't support a workflow:
1. Open Claude Code
2. Ask it to build the thing needed
3. Claude Code writes one-off scripts or modifies code
4. This takes 15-60 minutes per request
5. Sometimes the changes break other things

This is a developer workflow, not an operator workflow. The founder shouldn't need to modify code to query their own data.

## Target User & Narrowest Wedge

**Target user:** The founder (and eventually, other agency operators and advertisers using ViralTracker).

**Narrowest wedge:** Read-only data queries in Chainlit chat. "Tell me about ad X" and get publish date, purchases, ROAS, audience size in 5 seconds. No code changes, no page navigation, no one-off scripts.

## Constraints

- Chainlit already deployed and working with 9 agents / 70 tools
- PydanticAI agent framework with established thin-tool pattern
- Multi-tenant auth (Supabase), feature gating (FeatureKey), usage tracking (UsageTracker) all at service layer
- `scheduled_jobs` table has CHECK constraint on job_type — new job types need migration
- Gemini API rate limits prevent parallel pipeline execution (sequential is intentional)
- Must not break existing Streamlit UI or worker during migration
- Will eventually have other users — UX can't be embarrassing

## Premises

1. **Chat and UI are complementary.** The UI provides structured dashboards and visualization. Chat provides flexible composition and ad-hoc queries. Chat can eventually be embedded in UI pages for contextual interaction. New features default to chat-first unless they require rich visualization.
2. **Service layer is source of truth.** Chat tools are thin wrappers over the same services the UI calls. No new business logic in chat.
3. **Long-running ops go through the job queue.** Chat triggers jobs and gets push notifications. It doesn't run 15-minute pipelines inline.
4. **Multi-tenancy enforced at service layer.** Chat inherits by passing org/user context at session start.
5. **Progressive tool discovery solves scale.** PydanticAI's deferred tool loading loads core tools upfront, defers the rest for on-demand discovery.
6. **First value = ad-hoc data queries.** Not creative generation, not complex pipelines.

## Cross-Model Perspective

**Codex (independent cold read):**

Steelman: "ViralTracker should be a natural-language control plane over your existing service graph, not another UI layer. If it works, agency execution speed compounds because 'new workflow' becomes a prompt, not a sprint."

Most revealing quote: "Right now I want to know when this ad was published, how many purchases we've had." — The wedge is fast, trustworthy operational answers during live work, not autonomous campaign building.

Challenged premise: "Chat augments Streamlit, doesn't replace it." Codex argues keeping both first-class preserves the complexity that's breaking the founder. Evidence test: if >70% of daily tasks move to chat within 2-4 weeks, Streamlit usage will collapse naturally.

48-hour prototype: 10-15 read-only thin wrappers for highest-frequency queries, deferred tool loading, strict tenant context injection, response provenance (answer + source services + timestamps), job queue handoff for long tasks.

**Synthesis:** Both Claude and Codex agree the wedge is read-only data queries with thin wrappers and deferred loading. Codex's challenge led to a refined Premise #1: chat is designed to become the primary interface over time, with UI fading to dashboard-only. Codex's "provenance" idea (showing which services answered) included in the design.

## Approaches Considered

### Approach A: "Ops Copilot" — Read-first, expand later
Start with ~15 new read-only data query tools + wire up 11 existing services that lack tools. Use PydanticAI's deferred tool loading for the combined ~85+ tool set. Add job queueing + push notifications. Expand to write operations in Wave 2.

Effort: M (CC: ~5-7 days)
Risk: Low
Pros: Immediately useful, zero risk to existing UI, provenance builds trust
Cons: Embedded chat deferred, write operations come later, two interfaces for a while

### Approach B: "Contextual Chat" — Embedded from day one
Build a chat widget embedded in Streamlit pages via st.components. Passes page context (brand, ad IDs, filters) as conversation metadata. Standalone Chainlit also works.

Effort: L (CC: ~7-10 days)
Risk: Medium
Pros: "Ask about what I'm seeing" from day one, natural UX
Cons: Streamlit iframe quirks, more moving parts, page-context protocol needed

### Approach C: "API-first, UI-agnostic"
Expand FastAPI layer to expose all services as REST endpoints. New features and chat consume the API. Existing Streamlit pages migrate incrementally (not big-bang).

Effort: XL (CC: ~2-3 weeks)
Risk: Medium
Pros: Cleanest architecture, future React migration trivial, partner integrations possible
Cons: Largest upfront investment, delays "use it tomorrow" moment

## Recommended Approach

**A → C → Next.js → MCP.** Ship the Ops Copilot (Approach A) first. Use it for real client work. Build the API foundation (Approach C) informed by usage data. Then port high-pain Streamlit pages to Next.js. Finally, expose services as MCP tools for AI-native access from Claude Code, Claude Desktop, and other clients.

Embedded chat (Approach B) becomes trivial after C — once the API exists, a chat widget on any page (Streamlit or Next.js) is just a thin client.

### Service Coverage by Phase

| Phase | What changes | Access surfaces |
|-------|-------------|-----------------|
| Current state | 20 services with agent tools | Chainlit chat |
| Phase 1 (Ops Copilot) | +11 service wirings + ~15 new query tools (~46 total) | Chainlit chat |
| Phase 2 (High-value) | +15-20 services (competitor, SEO, planning) (~65 total) | Chainlit chat |
| Phase 3 (API foundation) | All 82 services get REST endpoints | Chainlit + API |
| Phase 4 (Next.js) | High-pain pages ported off Streamlit | Chainlit + API + Next.js |
| Phase 5 (MCP server) | Core tools exposed as MCP | Chainlit + API + Next.js + MCP |

Not all 82 services need chat tools. Some (like `scorer_weight_learning`, `template_scoring`) are internal machinery. The API makes them accessible; chat tools are only added when conversational access makes sense.

### Risk Mitigation

- **Feature flag:** New Ops Copilot tools are registered behind a `CHAT_OPS_ENABLED` env var. Set to `false` to disable all new tools without redeploying. Existing 70 tools are unaffected.
- **Rollback:** If the Ops Copilot causes issues, flip the flag. Chainlit falls back to the existing 9 agents / 70 tools.
- **Duplicate job prevention:** `queue_job` checks for an active job of the same `job_type` + `brand_id` created in the last 5 minutes. If found, confirms with the user: "A {job_type} job for {brand} was queued 3 minutes ago (status: running). Queue another?"

### Pre-Phase Validation Spike (Day 0 — 2-4 hours)

Before committing to implementation, verify two load-bearing assumptions:

1. **Chainlit background push:** Create a test background task in `chainlit_app/app.py` that sends a `cl.Message` after 10 seconds. Verify it reaches the correct user session via WebSocket. If this fails, fall back to polling-based status checks (user asks "is my job done?" instead of getting notified).

2. **PydanticAI deferred tool loading:** Verify the installed PydanticAI version supports `defer_loading=True` on tools or `.defer_loading()` on toolsets. Create a test agent with 5 deferred tools and verify the search/discovery mechanism works. If unavailable, fall back to a manual tool-routing agent that selects the right specialist agent based on query classification (the existing orchestrator pattern).

### Phase 1: Ops Copilot (Wave 1 — 5-7 days)

**Chainlit Auth Flow:**
- User authenticates via Supabase (existing `chainlit_app/auth.py` handles this)
- On `@cl.on_chat_start`, extract `user_id` and `organization_id` from the Supabase session
- Create `AgentDependencies.create(organization_id=..., user_id=...)` and store in `cl.user_session`
- Session expiry: if the Supabase token expires mid-conversation, the next tool call will fail with an auth error. Surface this as "Session expired, please refresh" rather than a cryptic error.

**New core tools (~15, loaded upfront):**
- `query_ad_performance(ad_id)` — publish date, purchases, ROAS, spend, audience size
- `query_top_ads(brand_id, metric, date_range, limit)` — top N ads by any metric
- `query_aggregate_metrics(brand_id, group_by, date_range)` — cost per ATC by awareness, etc.
- `queue_job(job_type, parameters)` — generic job trigger for all 8 existing job types (with duplicate detection)
- `check_job_status(job_id)` — job progress and results
- `list_recent_jobs(brand_id, status)` — what's running, what completed, what failed

**Wire up 11 services already in AgentDeps that lack tools:**
BrandResearchService, TemplateQueueService, ProductURLService, ContentPipelineService, RedditSentimentService, ProductContextService, BeliefAnalysisService, MetaAdsService, SEOProjectService, DocService, UsageTracker

**Deferred Tool Discovery:**

When the agent's loaded toolset doesn't cover a query, a `discover_tools(keyword)` meta-tool searches deferred tool descriptions:

- **Search method:** Substring match against tool names and docstrings. If PydanticAI's built-in discovery supports embedding-based search, use that. Otherwise, simple keyword matching is good enough for v1 (85 tools is small enough that keyword recall is acceptable).
- **Scope of loading:** Tools discovered in a turn remain available for the rest of the session. This prevents re-discovery on follow-up questions.
- **Max per call:** Load at most 5 tools per discovery to avoid context bloat. If more than 5 match, return the top 5 by relevance with a note: "Found {N} matching tools, showing top 5. Refine your query for more specific results."
- **Zero results:** If no tools match, respond: "I don't have a tool for that yet. You may need to use the Streamlit UI for now."
- **Expected totals:** ~85 tools (70 existing + ~15 new), with ~15 loaded upfront and ~70 deferred.

**Usage tracking:**
All new tools that invoke AI-backed services (Gemini, Claude, OpenAI) must wire up `UsageTracker` via `ctx.deps.usage_tracker`, consistent with existing tools. Read-only database queries do not need usage tracking.

**Push notification background task:**
- Scope: per-session asyncio task, started on `@cl.on_chat_start`, cancelled on session end
- Polling interval: every 30 seconds (one SELECT query against `scheduled_jobs` where `user_id` matches and `status` changed since last poll)
- When a job completes: `await cl.Message(content="Job completed: {job_type} for {brand_name}. {summary}").send()`
- When multiple jobs complete simultaneously: send one message per job
- If the user closes the tab: the background task dies with the session. Job results are in the DB regardless.

**Job failure handling in chat:**
- Job failed: "Job {job_id} failed: {error_message}. The job has been rescheduled for {next_run_at} per the retry policy."
- Job stuck (no status change for >2x expected duration): "Job {job_id} appears stuck — running for {duration}."
- Estimated completion time: v1 uses hardcoded estimates per job type (e.g., "template scrape ~5 min, ad creation ~10 min"). Refine with historical p50 data in Phase 2.

**Response format:**
- Answer + source services used + timestamps (provenance)
- For job triggers: job ID + estimated completion + "I'll notify you when it's done"

### Phase 2: High-Value Services (Wave 2 — 5-7 days)

- Competitor intel, image analysis, winner DNA, pattern discovery
- Planning and template recommendations
- Brand research and content pipeline triggers
- SEO keyword analysis and linking opportunities
- Ad creation workflows (complete_ad_workflow already has a tool)
- Refine provenance formatting based on Phase 1 usage
- Add historical completion time estimates if hardcoded ones are inaccurate

### Phase 3: API Foundation (Approach C — 2-3 weeks)

- Expand FastAPI with REST endpoints for services that have chat tools
- Auth middleware (JWT from Supabase, org context extraction)
- Rate limiting per org
- OpenAPI spec auto-generated from service signatures
- **New features** consume the API from both Chainlit and Streamlit
- **Existing Streamlit pages migrate incrementally** — not a big-bang rewrite. Pages are updated to call the API when they're next touched for bug fixes or feature additions. This is a gradual migration, not a dedicated phase.
- Internal-only services (template_scoring, scorer_weight_learning, etc.) get API endpoints but not chat tools

## Open Questions

1. **Embedded chat protocol:** When we add embedded chat on pages, what context does each page pass? (current brand_id, selected ad IDs, active filters, page name?) Defer to post-Phase 3.
2. **Tool naming conventions:** With 85+ tools, naming matters for progressive discovery. Adopt `{domain}_{action}` pattern? (e.g., `ads_query_performance`, `seo_find_links`, `brand_research_analyze`)
3. **Push notification backup:** If the user closes the Chainlit tab, job results are in the DB. Should we add Slack notification as backup? Defer to Phase 2.
4. **Streaming responses:** Start with complete responses. Add streaming if query latency exceeds 5 seconds.

## Success Criteria

- **Day 0:** Validation spike passes (Chainlit push works, deferred loading works)
- **Week 1:** Can ask "tell me about ad X" and get publish date, purchases, ROAS in <10 seconds
- **Week 1:** Can queue any of the 8 existing job types from chat and get notified on completion
- **Week 2:** 15+ read-only query tools working with provenance. High-value write services begin.
- **Week 3-4:** Phase 2 complete. Ad creation, SEO, competitor intel accessible from chat.
- **Month 2-3:** API foundation started. New features are API-first.
- **Month 3-4:** Next.js scaffold live. First batch of high-pain pages ported.
- **Month 4-5:** MCP server live. Claude Code can call ViralTracker services directly.
- **Ongoing:** New features built in Next.js + chat-first. Streamlit pages stop growing.

## Dependencies

- PydanticAI deferred tool loading (verified in Day 0 spike)
- Chainlit WebSocket push from background tasks (verified in Day 0 spike)
- No new job types needed for Wave 1 (reuses existing 8 job types)
- If new job types are added in Wave 2+, need migration to update scheduled_jobs CHECK constraint

## Reviewer Concerns

Issues identified across 2 rounds of adversarial review that are acknowledged but not fully resolved in this design:

1. **Discovery recall at scale (F1):** Keyword-based tool discovery may have poor recall when natural-language queries don't match tool names. Mitigation: v1 uses simple keyword matching which is adequate for 85 tools. If recall is poor in practice, upgrade to embedding-based search over tool descriptions.

2. **Phase 3 scope ambiguity (S1):** "Expand FastAPI for all services" is a large undertaking. Mitigation: Phase 3 is scoped to new-features-first with incremental migration. Full scope will be defined in a separate design doc when Phase 2 is complete and we have usage data.

## Phase 4: Next.js Frontend (Post-Phase 3)

Once the API foundation is in place, build new pages in Next.js instead of Streamlit. The API built for chat is the same API the frontend calls.

**Sequence:**
1. Next.js scaffold: auth flow (Supabase JWT), layout, brand selector, shared nav (~1-2 weeks CC)
2. Port high-pain pages first: Ad Creator, Scheduler, Experiments — the ones with heaviest session_state and worst async hacks (~1-2 weeks per batch of ~5 pages)
3. Ongoing: new features built in Next.js, existing Streamlit pages ported when next touched
4. Streamlit atrophies naturally — stop investing in it, keep it running for pages that haven't been ported

**Why this works after Phase 3:** The API handles auth, rate limiting, and org context. Next.js pages are thin consumers. No business logic in the frontend. The same API serves Chainlit chat, Next.js pages, and MCP tools.

**What goes away for free:**
- `nest_asyncio.apply()` hack — Next.js doesn't have Streamlit's event loop conflict
- Auth logout race condition — no more cookie iframe timing races
- 2,983 `session_state` references — React state management replaces Streamlit's full-rerun model
- Services recreated on every rerun — API handles service lifecycle, frontend just calls endpoints
- Thread-local DB clients — API is properly async, no Streamlit threading workarounds

**What stays in Streamlit (potentially forever):**
- Pages that are simple dashboards with minimal interactivity
- Admin/system pages that only you use
- Anything not worth porting

## Phase 5: MCP Server

Expose ViralTracker services as MCP (Model Context Protocol) tools. This means any MCP-compatible client — Claude Code, Claude Desktop, Cursor, or any future AI agent — can call your services directly.

**Why this matters:** Your current workaround is opening Claude Code and asking it to write one-off scripts. With MCP, Claude Code calls your services directly. "What's the ROAS on ad X?" just works, no scripts needed. This is the natural endpoint of "make services accessible from anywhere."

**Architecture:**
- PydanticAI has built-in MCP server support — the same tool definitions used by Chainlit agents can be exposed as MCP tools
- Auth: MCP server authenticates via API key or JWT, same as the FastAPI layer
- Org context: passed as MCP resource metadata or conversation context
- Usage tracking: enforced at the service layer (same as chat and API)

**Scope:**
- Start with the same ~15 core query tools from Phase 1 (ad performance, top ads, job queue)
- Expand to match whatever's available in Chainlit
- Internal-only services (scoring, template evaluation) stay API-only, not MCP

**Effort:** Small (~2-3 days CC) once Phase 3's API exists. The tools are already defined; MCP is just another transport layer.

**Distribution:**
- Local MCP server for the founder (runs alongside the app, Claude Code connects via stdio or SSE)
- Remote MCP server for other users (hosted, authenticated, connects via SSE)

This means the "interface" for ViralTracker becomes: Chainlit chat for conversational workflows, Next.js for structured dashboards, MCP for AI-native access from any client. All three consume the same API, all three respect the same auth/tenancy/usage rules.
