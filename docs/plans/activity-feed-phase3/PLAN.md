## Problem Statement

The Activity Feed (shipped Phase 2) is the operating console for ViralTracker. It shows what the system has been doing across all brands and job types. But today, every event card is text-only: a severity icon, timestamp, title, and metadata caption. When an `ads_generated` event fires saying "8 ads generated," the founder has to click through to another page just to see what those 8 ads look like. Same for `templates_scraped`: "Scraped 12 new templates" tells you a count but not what was scraped.

The founder uses the feed as a triage layer: glance at output, decide whether to investigate. Text-only cards force an unnecessary page navigation for every event that produces visual output. Rich media cards show enough to decide without leaving the feed.

## Demand Evidence

The founder IS the user. They use the Activity Feed daily as their first stop. Quote: "I'd like to click through when I think I need to. Like if I want to see all the ads generated I might want to click through, or if something looks off or fishy I'll click through."

This confirms the thumbnails serve as a decision tool, not decoration. The user wants to see enough to know whether the output looks right, then optionally dive deeper. The "+N more" indicator and click-through behavior are essential, not nice-to-have.

## Status Quo

Current workflow when an `ads_generated` event appears:
1. See "8 ads generated" in the Activity Feed (text only)
2. Click "View" button which navigates to Ad Scheduler page
3. Find the relevant job run
4. Expand it to see the generated creatives
5. Decide whether the output looks reasonable
6. Navigate back to the Activity Feed

This 5-step process happens multiple times per day across multiple brands. The media grid collapses steps 2-5 into a glance.

## Target User & Narrowest Wedge

**User:** Solo founder/operator managing multiple brands via ViralTracker. Checks the Activity Feed as first action each day. Needs to triage system output quickly.

**Narrowest wedge:** Two event types only:
- `ads_generated` — thumbnails of generated ad creatives from `generated_ads.storage_path` (Supabase Storage, bucket: `generated-ads`)
- `templates_scraped` — thumbnails of scraped competitor ads from `scraped_ad_assets.storage_path` (Supabase Storage, bucket: `scraped-assets`)

Asset downloads and SEO article images deferred to Phase 3b.

## Constraints

1. **Streamlit rendering**: Must use `st.image()` or `st.markdown()` with HTML. Streamlit has limited CSS support but `st.markdown(unsafe_allow_html=True)` enables custom HTML grids. **Pre-implementation verification required**: test `st.markdown('<img src="https://your-supabase-url/..." />', unsafe_allow_html=True)` in a scratch page to confirm Streamlit renders external images in HTML.
2. **Facebook CDN URLs expire**: Must use Supabase Storage paths, never raw Facebook image URLs. Both `generated_ads` and `scraped_ad_assets` already store stable `storage_path` values.
3. **Progressive enhancement**: Events without `thumbnail_urls` in their `details` JSONB render exactly as they do today. No migration needed. No breaking change.
4. **Performance**: The Activity Feed loads 50 events per page. No additional DB queries at render time. All thumbnail URLs pre-computed at emit time.
5. **Partial results are fine**: If only 3 of 5 rows have `storage_path` populated, the grid renders with 3 images. The grid adapts to however many URLs are available (1-5).

## Premises

1. **Denormalize image URLs into `details` JSONB at emit time.** The worker already has access to storage paths. Storing 5-8 thumbnail URLs when the event fires means zero additional queries at render time.
2. **Use Supabase Storage URLs, never raw external CDN URLs.** Facebook CDN URLs expire quickly. Both event types have stable `storage_path` columns in Supabase Storage.
3. **Start with `ads_generated` and `templates_scraped` only.** Asset downloads and SEO articles are Phase 3b.
4. **The image grid is a progressive enhancement.** Events without images render exactly as today. No migration for existing events.

## Approaches Considered

### Approach A: Denormalize at Emit Time (Chosen)

Worker looks up top 5 `storage_path` values when emitting events, constructs Supabase public URLs, stores them in `details.thumbnail_urls` array and `details.thumbnail_count` for total count. UI renders a CSS grid from these pre-computed URLs using `st.markdown(unsafe_allow_html=True)` with an `st.columns()` fallback.

- **Effort:** S-M
- **Risk:** Low
- **Pros:** Zero additional DB queries at page load; progressive enhancement (old events unchanged); changes to 2 files (worker + UI), though the templates_scraped path also requires collecting asset IDs in the scrape loop
- **Cons:** Existing events won't have thumbnails (acceptable given feed is 1 day old)
- **Reuses:** Existing `_emit_activity_event()`, `render_event_card()`, `storage_path` columns

### Approach B: Render-Time Lookup with Caching

UI detects media-capable event types, queries source tables using IDs in `details`, caches in `st.session_state`.

- **Effort:** M
- **Risk:** Medium
- **Pros:** Works for all existing events
- **Cons:** N+1 queries per page load (50 events x 1 query); complex cache invalidation; fragile coupling to source table schemas

### Approach C: Hybrid with Backfill Script

Same as A going forward, plus a one-time backfill script for historical events.

- **Effort:** M
- **Risk:** Low-Med
- **Pros:** All events (old and new) get thumbnails
- **Cons:** Extra script; backfill may miss events where source data was deleted

## Recommended Approach

**Approach A: Denormalize at Emit Time.**

The feed is 1 day old. There are almost no historical events worth backfilling. The simplest change wins: worker stores URLs at emit time, UI renders them. If backfill becomes desirable, it's trivial to add later.

### Implementation Details

#### Worker Changes (`scheduler_worker.py`)

**For `ads_generated` events (~line 1771):**

After ads are generated, before emitting the event, look up storage_paths for the generated ads. The query uses `ad_run_ids[:5]` with `limit(5)` to get up to 5 images total. This may return multiple images from the same ad_run or spread across runs... either is fine. The purpose is a quick visual check, not representative coverage of every template.

```python
thumbnail_urls = []
if ad_run_ids:
    try:
        thumb_result = db.table("generated_ads").select("storage_path").in_("final_status", ["approved", "flagged"]).in_(
            "ad_run_id", ad_run_ids[:5]
        ).limit(5).execute()
        thumbnail_urls = [
            f"{supabase_url}/storage/v1/object/public/generated-ads/{r['storage_path']}"
            for r in (thumb_result.data or []) if r.get("storage_path")
        ]
    except Exception:
        pass  # Non-blocking: thumbnails are best-effort
```

Add `thumbnail_urls` and `thumbnail_count` to the details dict.

**For `templates_scraped` events (~line 2662):**

**Important:** The variable `new_asset_ids` does not currently exist in the worker. The scrape loop tracks `new_count` (integer) but does not accumulate a list of new asset IDs. The implementation must add a `new_asset_ids = []` list before the scrape loop (~line 2578) and append each newly created asset's ID during the loop (where `new_count += 1`). This mirrors how `ad_run_ids` is accumulated in the ad creation flow.

After the loop, before emitting the event:

```python
thumbnail_urls = []
if new_asset_ids:
    try:
        thumb_result = db.table("scraped_ad_assets").select("storage_path").in_(
            "id", new_asset_ids[:5]
        ).eq("status", "downloaded").limit(5).execute()
        thumbnail_urls = [
            f"{supabase_url}/storage/v1/object/public/scraped-assets/{r['storage_path']}"
            for r in (thumb_result.data or []) if r.get("storage_path")
        ]
    except Exception:
        pass
```

Add `thumbnail_urls` and `thumbnail_count` to the details dict.

#### UI Changes (`00_📊_Activity_Feed.py`)

Add a `render_media_grid()` function that takes `thumbnail_urls`, `total_count`, and optional `link_page`. 

**Primary approach:** Use `st.markdown(unsafe_allow_html=True)` to render a CSS grid with hero + thumbnails + "+N" overlay.

**Fallback:** If Streamlit strips `<img>` tags or CSS grid renders poorly, fall back to `st.columns([2,1,1])` with `st.image()` calls, which is natively supported and guaranteed to work.

Call from `render_event_card()` when `details.thumbnail_urls` is present:

```python
# Rich media grid (Phase 3)
thumbnail_urls = details.get("thumbnail_urls", [])
thumbnail_count = details.get("thumbnail_count", 0)
if thumbnail_urls:
    render_media_grid(thumbnail_urls, thumbnail_count, link_page=event.get("link_page"))
```

5. **Update code snippets to store tuples**: Worker stores `[{"bucket": "generated-ads", "path": "abc/image.png"}, ...]` in `details.thumbnail_urls`. Public URLs constructed at render time in `render_media_grid()`. Both voices flagged this contradiction as critical.

7. **Images are non-clickable** (view-only): Streamlit `st.markdown(unsafe_allow_html=True)` cannot trigger `st.switch_page()` from HTML click events. The "View All" button remains the sole navigation mechanism. Remove `cursor: pointer` from wireframe.

8. **Add `loading="lazy"`** to all `<img>` tags: Prevents 250 simultaneous image requests on page load. Below-fold images load on scroll.

9. **Add ORDER BY to thumbnail query**: `ORDER BY created_at DESC` on both queries so the hero image is the most recent, not arbitrary.

10. **All-images-fail fallback**: If all images fire `onerror`, hide the entire `.media-grid` container. User sees text-only card instead of empty box.

### Eng Amendments (bugs in plan that must be fixed before implementation)

11. **CRITICAL: Remove `final_status` filter** — Ads are freshly created at emit time. Status is likely "pending", not "approved"/"flagged". Filter should be `.not_.is_("storage_path", "null")` instead.

12. **CRITICAL: Use `split("/", 1)` to parse storage_path** — Both `generated_ads.storage_path` and `scraped_ad_assets.storage_path` may include bucket prefix (e.g., `"generated-ads/WP-C3-..."` or just `"WP-C3-..."`). The existing codebase has a `split("/", 1)` guard for this mixed format. Tuple construction must handle both formats.

13. **HIGH: Add `asset_type='image'` filter** to templates_scraped query — `scraped_ad_assets` includes both images and videos. Without this filter, video assets produce broken thumbnails.

14. **HIGH: Convert UUID to string** in `new_asset_ids` accumulation — `asset_result['images']` returns UUID objects. Supabase `.in_("id", ...)` needs strings.

15. **HIGH: Use `get_public_url()` pattern** in UI renderer — Don't string-format Supabase URLs. Use existing `db.storage.from_(bucket).get_public_url(path)` pattern from `pages/27_Plan_Executor.py:249-268`.

16. **MEDIUM: Replace `except Exception: pass` with logging** — Add `logger.debug("Thumbnail lookup failed for event: %s", e)` to catch systematic failures.

17. **MEDIUM: Clarify `thumbnail_count`** — Should be total count of ads/assets generated (from the event, not from the thumbnail query). The `+N` badge shows `thumbnail_count - len(thumbnail_urls)`. If assets are later deleted, the count becomes stale, but this is acceptable for a solo-founder tool.
